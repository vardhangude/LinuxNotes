# 1. SSH (Secure Shell)

## Definition

SSH (Secure Shell) is a secure network protocol that allows administrators to remotely log in to Linux servers over an encrypted connection.

SSH replaced Telnet because Telnet sends usernames and passwords in plain text, whereas SSH encrypts all communication between the client and server.

Default Port:

```bash
22
```

---

## Why do we use SSH?

SSH is used because Linux servers are often deployed in remote environments such as AWS, Azure, Google Cloud, or corporate data centers. Instead of physically accessing the server, administrators securely connect using SSH.

Common use cases include:

- Remote server administration
- Executing commands
- Software installation
- Log analysis
- File transfers (using SCP/SFTP)
- Automation through scripts

---

## Interview Answer

> SSH stands for Secure Shell. It is a secure protocol used to remotely access Linux servers over an encrypted connection. It operates on port 22 by default and uses authentication methods such as passwords or SSH keys. In production, SSH is the standard method for securely managing servers, executing commands, deploying applications, and troubleshooting systems.

# 2. SSH Architecture

SSH communication consists of two components:

### SSH Client

The system initiating the connection.

Examples:

- Laptop
- Administrator workstation
- Jenkins server
- Bastion host

### SSH Server

The remote Linux server running the SSH daemon (`sshd`) that accepts incoming connections.

---

## Connection Flow

```text
SSH Client
      │
      │ TCP Port 22
      ▼
SSH Server (sshd)
      │
Authentication
      │
Encrypted Session
```

# 3. SSH Authentication

SSH supports two common authentication methods:

### 1. Password Authentication

The user enters a username and password.

Example:

```bash
ssh user@192.168.1.100
```

The server prompts:

```text
Password:
```

---

### Advantages

- Easy to use
- Suitable for lab environments

### Disadvantages

- Vulnerable to brute-force attacks
- Password management overhead
- Less secure than SSH keys

---

### 2. Key-Based Authentication

The client generates a public/private key pair.

The public key is stored on the server, while the private key remains securely on the client.

During login, the server verifies the client's private key without transmitting passwords.

---

## Why Key-Based Authentication?

- More secure
- No password prompts
- Enables automation
- Used by DevOps tools like Ansible, Jenkins, and Git

---

## Interview Answer

> In production, I prefer SSH key-based authentication because it provides stronger security than passwords. The client authenticates using a private key while the server stores the corresponding public key. This approach eliminates password transmission and is widely used for automation.

---

# 4. Generating SSH Keys

Generate a new key pair:

```bash
ssh-keygen
```

Common algorithm:

```bash
ssh-keygen -t rsa -b 4096
```

Or with ED25519:

```bash
ssh-keygen -t ed25519
```

---

Generated files:

Private key:

```text
~/.ssh/id_rsa
```

Public key:

```text
~/.ssh/id_rsa.pub
```

Never share the private key.

---

# 5. Copying Public Key to Server

```bash
ssh-copy-id user@server-ip
```

Example:

```bash
ssh-copy-id ec2-user@192.168.1.50
```

After copying:

```bash
ssh ec2-user@192.168.1.50
```

No password will be required.

---

## Manual Method

Append the public key to:

```text
~/.ssh/authorized_keys
```

Permissions:

```bash
chmod 700 ~/.ssh

chmod 600 ~/.ssh/authorized_keys
```

# 6. SSH Configuration File

Server configuration:

```text
/etc/ssh/sshd_config
```

After making changes:

```bash
sudo systemctl restart sshd
```

---

## Common Configuration Options

### Change SSH Port

Default:

```text
Port 22
```

Example:

```text
Port 2222
```

---

### Disable Root Login

```text
PermitRootLogin no
```

---

### Disable Password Authentication

```text
PasswordAuthentication no
```

---

### Enable Public Key Authentication

```text
PubkeyAuthentication yes
```

---

### Restrict Specific Users

```text
AllowUsers admin devops
```

---

## Why Modify sshd_config?

- Improve security
- Prevent unauthorized access
- Enforce key-based authentication
- Reduce brute-force attacks

# 7. Managing SSH Service

Start:

```bash
sudo systemctl start sshd
```

Stop:

```bash
sudo systemctl stop sshd
```

Restart:

```bash
sudo systemctl restart sshd
```

Reload configuration:

```bash
sudo systemctl reload sshd
```

Check status:

```bash
systemctl status sshd
```

Enable on boot:

```bash
systemctl enable sshd
```

# 8. Production Scenario

A developer reports:

> "I cannot SSH into the production server."

### Step 1

Verify server reachability.

```bash
ping server-ip
```

---

### Step 2

Verify SSH port.

```bash
ss -tulpn | grep ssh
```

---

### Step 3

Check service.

```bash
systemctl status sshd
```

---

### Step 4

Review logs.

RHEL:

```bash
journalctl -u sshd
```

or

```bash
cat /var/log/secure
```

---

### Step 5

Verify firewall.

```bash
firewall-cmd --list-all
```

---

### Step 6

Confirm port accessibility.

```bash
nc -zv server-ip 22
```

or

```bash
telnet server-ip 22
```

# 9. SSH Troubleshooting

## Permission Denied

Possible causes:

- Wrong password
- Incorrect private key
- Incorrect file permissions
- Public key missing

Check:

```bash
ls -ld ~/.ssh

ls -l ~/.ssh
```

---

## Connection Refused

Possible reasons:

- sshd service stopped
- Wrong port
- Firewall blocking SSH

---

## Host Key Changed

Error:

```text
WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!
```

Solution:

```bash
ssh-keygen -R server-ip
```

---

## Authentication Failure

Use verbose mode:

```bash
ssh -v user@server
```

For more debugging:

```bash
ssh -vvv user@server
```

---

# 10. Best Practices

- Disable root login.
- Use SSH keys instead of passwords.
- Disable password authentication where possible.
- Restrict users with `AllowUsers`.
- Rotate SSH keys periodically.
- Keep OpenSSH packages updated.
- Monitor authentication logs.
- Use a bastion host for production environments.
- Avoid sharing private keys.
- Backup SSH configuration before changes.
