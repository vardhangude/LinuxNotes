# 1. Linux Permission Model

Every file has three permission sets:

-   User (Owner)
-   Group
-   Others

```{=html}

```
    -rwxr-xr--

  Section   Meaning
  --------- -----------
  \-        File type
  rwx       Owner
  r-x       Group
  r--       Others

### Permission Values

  Permission      Value
  ------------- -------
  Read (r)            4
  Write (w)           2
  Execute (x)         1

Examples

    Octal Meaning
  ------- ---------------------
      755 rwx r-x r-x
      750 rwx r-x ---
      644 rw- r-- r--
      600 rw-------
      777 Avoid in production

# 2. chmod

Change file permissions.

## Numeric

``` bash
chmod 755 script.sh
chmod 644 app.conf
chmod -R 755 /opt/app
```

## Symbolic

``` bash
chmod u+x script.sh
chmod g-w file
chmod o-r file
chmod a+x script.sh
```

### Production Example

Deployment script is not executable.

``` bash
chmod +x deploy.sh
```

### Interview

**Why is 777 discouraged?**

It gives everyone write access, increasing the risk of accidental or
malicious changes.

------------------------------------------------------------------------

# 3. Ownership

## chown

``` bash
chown nginx index.html
chown nginx:nginx index.html
chown -R appuser:appgroup /opt/app
```

## chgrp

``` bash
chgrp developers app.log
```

### Difference

  Command   Purpose
  --------- -------------
  chmod     Permissions
  chown     Owner
  chgrp     Group only

### Troubleshooting

``` bash
ls -l file
id appuser
```

Verify owner, group and permissions before changing anything.

# 4. Special Permissions

## SUID (4000)

Runs executable with owner's privileges.

``` bash
chmod 4755 program
```

Example:

``` bash
ls -l /usr/bin/passwd
```

Why?

Normal users can update their own password because the command runs with
root privileges.

------------------------------------------------------------------------

## SGID (2000)

Directories:

New files inherit the directory's group.

``` bash
chmod 2775 /shared
```

Useful for development teams sharing files.

------------------------------------------------------------------------

## Sticky Bit (1000)

Only file owner or root can delete files.

``` bash
chmod 1777 /shared
```

Common example:

``` bash
ls -ld /tmp
```

------------------------------------------------------------------------

# 5. Umask

Defines default permissions for newly created files/directories.

Check:

``` bash
umask
```

Set:

``` bash
umask 022
```

Typical defaults:

    Umask File   Directory
  ------- ------ -----------
      022 644    755
      027 640    750
      077 600    700

Interview:

Why use umask?

To enforce secure default permissions without changing every file
manually.

------------------------------------------------------------------------

# 6. Access Control Lists (ACL)

ACL provides permissions beyond owner/group/others.

When to use:

-   One user needs access.
-   Multiple teams share data.
-   Avoid changing ownership.

## Commands

View ACL

``` bash
getfacl project.txt
```

Grant access

``` bash
setfacl -m u:john:rwx project.txt
```

Grant group access

``` bash
setfacl -m g:developers:r-x project.txt
```

Default ACL

``` bash
setfacl -d -m g:developers:rwx /project
```

Remove ACL

``` bash
setfacl -b project.txt
```

Recursive

``` bash
setfacl -R -m u:john:rwx /project
```

### ACL Mask

ACL mask limits maximum effective permissions.

Always verify using:

``` bash
getfacl filename
```

### Detect ACL

``` bash
ls -l
```

A trailing **+** means ACL entries exist.

Example

    -rw-rw-r--+

# Production Scenarios

## Scenario 1

Application reports **Permission denied**.

Checklist

``` bash
ls -l app.log
id appuser
getfacl app.log
```

Check:

-   Owner
-   Group
-   chmod
-   ACL
-   Parent directory permissions

------------------------------------------------------------------------

## Scenario 2

Shared project folder.

Requirement:

-   Developers should edit files.
-   Others should have read-only access.

Solution:

-   Create group.
-   Assign ownership.
-   Apply SGID.
-   Configure Default ACL.

------------------------------------------------------------------------

# Best Practices

-   Never use `chmod 777` unless absolutely required.
-   Prefer groups over ACL where practical.
-   Use ACL for exceptions.
-   Apply SGID to shared directories.
-   Use umask to enforce secure defaults.
-   Verify with `ls -l` and `getfacl` after changes.
